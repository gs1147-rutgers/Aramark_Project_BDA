Dashboard Overview & Architecture
What the dashboard is, what it does, and how it's structured

The ARAMARK Spend Prediction Dashboard is a fully interactive Plotly-Dash web application hosted at http://127.0.0.1:8051. It does three things in one unified interface:

  1. Retrospective analysis — explores the full 2025 Aramark spend dataset (43M+ SRF transactions) by state, segment, category, and time.
  2. Event attribution — overlays a curated calendar of 100+ major US events (2025 + 2026 projections) and quantifies their impact on hospitality procurement demand.
  3. Forward prediction — uses four trained ML models, weighted by cross-validated accuracy, to forecast monthly spend for all states across H1 2026 (Jan–Jun), with Apr–Jun being the true forecast window.

Dataset — Andrew Meszaros SRF 2025
Where the data comes from, what it contains, how it was prepared

Primary Dataset: SRF CSV (Spend Request Form)

The raw dataset Andrew_Meszaros_SRF_2026-04-01-0936.csv is a Spend Request Form extract covering Aramark's procurement activity across all US states and territories. Each row represents one purchase transaction.

FieldDescriptionUsed for
Transaction date / Year-MonthPurchase timestamp aggregated to YYYYMMTime-series training, lag features
StateUS state abbreviation (CA, TX, FL…)Geographic segmentation, join with reference data
Spend Random FactorAnonymised dollar spend per transactionTarget variable for prediction
Category Level 1Top-level procurement category (Food, Chemicals, Linen…)Category DNA analysis, eComm breakdown
Segment codeAramark business segment (MS-100000 … MS-100115)Segment deep-dive forecasts
eCommerce StatusACTIVE / INACTIVE / PENDING / ACTIVATEDeCommerce adoption analysis
Property roomsNumber of rooms at the propertySpend-per-room efficiency metric

Pre-Aggregation Pipeline

Because 43M rows cannot be queried at interactive speed, the raw CSV was pre-aggregated offline by feature_engineering.py into nine Parquet files:

  • monthly_state_agg.csv — (state, year_month, spend) — backbone of all time-series forecasting
  • state_spend_agg.csv — total + average spend, transaction count per state
  • seg_state_monthly.parquet — segment × state × month spend — powers the Segment Deep-Dive tab
  • seg_cat_dna.parquet — category % share per segment — powers the Category DNA radar
  • national_cat.parquet — national average category mix for benchmarking
  • spr_agg.parquet + spr_seg.parquet — spend-per-room by size bucket
  • ecomm_seg.parquet, ecomm_cat.parquet, ecomm_state.parquet — eCommerce status breakdown

Supplementary Reference Data

Because spend alone doesn't explain why some states over-perform, we enriched every state with hand-curated reference data from public sources:

  • Bureau of Economic Analysis (BEA) — state GDP ($B)
  • US Census Bureau — state population (millions)
  • US Travel Association / AHLA — tourism revenue ($B), hotel count, hotel rooms (thousands)
  • us_major_events_2026.xlsx / USA_Major_Events_2026.xlsx — verified 2026 event calendar with economic impact estimates sourced from PredictHQ, Bay Area Host Committee, Indy 500 IMS reports, Mardi Gras 2026 economic study
Important caveat on "Spend Random Factor": The spend values are a randomised proxy for actual dollar amounts (for data privacy). Correlations and directional trends are real; absolute dollar values should be treated as ordinal, not literal. The ML model learns relative patterns — seasonality, event lift, geographic structure — which remain valid even with randomised scaling.

Training / Test Split Strategy

  • Training set: All 2025 months (Jan–Dec 2025) for all available states. ~600 observations.
  • Evaluation: 5-fold cross-validation on the training set (not a held-out test set, since we're forecasting into the future).
  • Forecast window: Jan–Jun 2026. Jan–Mar 2026 use injected "actuals" (estimated from event outcomes) to prime lag features; Apr–Jun 2026 are the true ML predictions.

ML Framework — 4 Models + Weighted Ensemble
Model selection rationale, feature engineering, ensemble strategy, and prediction corridor

Feature Engineering — 16 Dimensions

Each training observation is one (state, year_month) pair encoded into 16 features:

FeatureTypeRationale
log_populationEconomicLog-scaled to reduce California/Texas dominance. More people → more hospitality transactions.
log_gdpEconomicState GDP ($B, log). Largest single correlator with total spend (r ≈ 0.87 in Pearson analysis).
log_tourismEconomicTourism revenue signals visitor-driven demand — especially important for FL, TN, NC.
log_hotel_roomsEconomicHotel room inventory is a proxy for Aramark contract coverage (cleaning, F&B, linen).
month_sin / month_cosSeasonalCircular encoding of month (avoids December=12 ≠ January=1 discontinuity). Captures U-shaped seasonality.
event_scoreEventAggregate event impact (0–200+) using diminishing returns: primary + 50% secondary + 25% tertiary…
event_countEventNumber of distinct events in that state-month — measures breadth of demand pressure.
max_event_impactEventSingle largest event score (0–100). Differentiates a Super Bowl month from a minor-events month.
aramark_boost_MEventResearch-backed dollar estimate ($M) of Aramark's 5-channel capture: venue concessions + hotel F&B + airport dining + convention catering + institutional spillover.
lag1_logspendLagLog-spend from previous month. Most powerful single feature — spend is auto-correlated.
lag2_logspendLagLog-spend 2 months ago.
lag3_logspendLagLog-spend 3 months ago.
roll3_logspendLag3-month rolling average of log-spend. Smooths short-term volatility.
year_offsetTemporalSet to 0 for all observations — intentionally disabled to prevent out-of-distribution extrapolation on small states.
monthTemporalRaw month integer (1–12) as a linear feature, complementing the circular sin/cos encoding.
Why log-transform the target? Spend has a heavy right tail (Hawaii has extreme per-transaction values; California dwarfs Vermont). Predicting log(1+spend) and back-transforming with expm1() ensures the model isn't dominated by large-state outliers and gives proportionally equal training weight to all states.

The Four Models — Why Each Was Chosen

ModelWhy IncludedStrengthsWeaknesses
Gradient Boosting (GBM)
n_estimators=300, depth=4, lr=0.05		 Industry standard on structured tabular data. Typically wins Kaggle competitions on similar problems. Handles non-linearities (event spikes), feature interactions (events × tourism revenue), robust to outliers Can overfit on small datasets; slow to train vs RF
Random Forest
n_estimators=300, depth=8		 Ensemble baseline — high variance reduction through bagging. Good sanity-check against GBM. Naturally resistant to overfitting; fast; interpretable via feature importance Tends to under-predict extremes (regression to the mean)
Ridge Regression
alpha=1.0		 Linear baseline. If the data is genuinely linear, Ridge should compete; if GBM wins by a large margin, non-linearity is confirmed. Fast, interpretable, stable coefficients; good for regions with few events Cannot capture non-linear event spikes or interaction effects
SVR (RBF kernel)
C=10.0, ε=0.05, γ=scale		 Non-linear kernel captures smooth curves. Segment-level forecasting uses SVR as primary model. Good at capturing smooth cyclical patterns; regularised through ε-insensitive loss Computationally expensive; hard to interpret; sensitive to scaling

Ensemble Strategy — Inverse-RMSE Weighting

Instead of picking a single winner, we combine all four using inverse-RMSE weighted averaging:

weight(model) = (1 / CV_RMSE(model)) / Σ(1 / CV_RMSE)
This means models with lower prediction error get proportionally more influence in the final ensemble forecast. The ensemble prediction band (shaded area on charts) spans from the minimum to maximum single-model prediction, giving a visual confidence interval.

Prediction Corridor — Preventing Extrapolation Failures

Raw ML predictions on small states (Vermont, North Dakota) can extrapolate wildly when a major event is scored. A multi-anchor corridor prevents this:

  • Floor = max(65% of historical average, event-anchored floor if aramark_boost ≥ $3M, 90% of same month last year)
  • Ceiling = 155%–220% of baseline (higher for large confirmed events like Super Bowl or FIFA WC)
  • Organic cap: For months with no real events (event_score < 30), cap at +15% of same month last year to prevent phantom growth.

Event Score vs Predicted Spend Lift — Bubble Scatter
Tab: 🗓 Event Heatmap · Bottom chart · Static
📊 Dataset Used
Computed from STATE_PREDS (ML forecasts) + event_score() for Apr–Jun 2026 months. X-axis = total 3-month event score; Y-axis = predicted spend lift vs Apr–Jun 2025 same period; bubble size = state GDP.
🤔 Why This Chart?
This is the validation chart for the entire model. If events truly drive spend, states with higher event scores should have higher predicted lifts. 🔍 How to Analyze
Upper-right quadrant: High events AND high lift — states like TX, CA, NJ, IN during FIFA/Indy season. These are the priority opportunity states.
Lower-left: Few events, modest lift — organic growth states (e.g. ND, VT).
Upper-left outliers: High lift but low event score — could indicate base effects (a low 2025 baseline inflating % growth) or structural market expansion not captured in event calendar.

💡 Approach
The trend line is a simple OLS regression across states. A positive slope with R² > 0.4 validates that the event calendar is genuinely predictive — not decorative. This chart is the "so what?" slide that ties the entire analysis together for executives.

Segment Forecast Chart (SVR + Ridge Ensemble)
Tab: 📊 Segment Deep-Dive · Interactive (segment dropdown + state filter)
📊 Dataset Used
seg_state_monthly.parquet (segment × state × month spend). Joined to national aggregate seg_monthly_raw when "All States" is selected.
🤔 Why SVR + Ridge Here (Not GBM)?
At the segment level, the training set per (segment, state) is much smaller — sometimes only 8–10 months of data. GBM with 300 trees overfits dramatically on such small samples. SVR (RBF) with regularisation (C=5) and Ridge offer smoother, more stable generalisation on sparse series. The segment-level model uses only 7 features (sin/cos, 3 lags, rolling mean, month) — a deliberately minimal feature set to avoid overfitting.
🔍 How to Analyze This Graph
Red solid line: 2025 actuals for the selected segment (Jan–Dec 2025).
Red dashed line (diamonds): H1 2026 forecast.
Shaded band: min(SVR, Ridge) to max(SVR, Ridge) — confidence range.
KPI panel (right side): H2 2025 actual vs H1 2026 forecast with growth %. This is the headline number for segment managers.
Look for segments where the forecast line sharply diverges from the seasonal baseline — that divergence is driven by the lag features picking up recent momentum or Q4 2025 trend.
💡 Inspiration
Segment managers at Aramark don't care about "all states combined" — they manage specific verticals (Healthcare Dining, Sports & Entertainment, Hotel/Lodging). This tab gives each segment owner a personalised forecast they can drill into by state. The state filter is the key UX innovation: "Show me Hospitality Large segment in Florida only" isolates the Miami/Orlando hotel F&B channel for that manager.

Tab 6 — Category DNA Fingerprint
What categories does each segment buy? Radar, treemap, and over/under-index analysis
12
Category DNA Radar — Segment vs National Average
Tab: 🧬 Category DNA · Left panel · Interactive (segment dropdown)
📊 Dataset Used
seg_cat_dna.parquet — pre-computed category % share per segment. national_cat.parquet — national average category mix as the benchmark polygon. Each axis on the radar = one procurement category (Food, Chemicals, Linen, Equipment, etc.).
🤔 Why a Radar Chart?
A radar (spider) chart is optimal for comparing multi-dimensional category profiles because: (1) deviations from the benchmark (grey dotted polygon) are immediately visible as spikes or gaps, (2) the overall "shape" of the polygon becomes an intuitive fingerprint — wide = diverse category mix, narrow spike = concentrated spend in one category, (3) comparison of two overlapping polygons (segment vs national) requires no legend decoding.
🔍 How to Analyze
Blue filled polygon: Selected segment's category % share.
Grey dotted polygon: National average — the "typical" Aramark customer.
Where blue exceeds grey: Segment over-indexes in that category (opportunity for category managers to negotiate volume deals).
Where grey exceeds blue: Segment under-indexes (untapped cross-sell opportunity or category not relevant).

Example: Healthcare Dining (MS-100001) should spike heavily on Food & Beverages and Chemicals (cleaning/sanitation) but have near-zero on Linen/Guest Amenities — a very different profile from Luxury Resort HI (MS-100026).
💡 Inspiration
Inspired by customer DNA profiling in retail analytics (e.g. Spotify's radar showing listening genre breakdown). Applied here to procurement to give category managers an instant "buying fingerprint" for each Aramark vertical. The term "DNA" was deliberately chosen to make the metaphor memorable for non-technical stakeholders.

Category Wallet Share Treemap
Tab: 🧬 Category DNA · Right panel · Interactive (segment dropdown)
🔍 How to Analyze
Each rectangle = one procurement category. Size = total spend. The largest rectangle is the dominant category for this segment. Labels show category name, dollar amount, and % of segment total. Colour gradient (blue → teal → green) by spend level. Zooming into the treemap reveals the exact portfolio composition at a glance — no pie chart sliver problem.
🤔 Why Treemap vs Bar Chart?
A horizontal bar chart with 15+ categories requires scrolling and makes area comparison difficult. A treemap uses area as the primary encoding — the human eye is very good at comparing rectangular areas side-by-side. 📊 How Models Were Evaluated
5-fold cross-validation on the 2025 training set. Metric: CV log-RMSE (Root Mean Squared Error in log-spend space). Log-RMSE was chosen over raw RMSE because raw RMSE would be dominated by California and Texas (very large absolute values), masking poor performance on smaller states. Log-space RMSE is interpretable as "proportional prediction error."
🏆 Which Model Wins?
Gradient Boosting consistently achieves the lowest CV log-RMSE. This aligns with the broader ML literature — GBM is the gold standard for structured tabular regression with mixed feature types (numeric, encoded categorical, time features). The winning model is highlighted in green on the RMSE bar chart.
📊 Feature Importance (GBM)
Expected ranking:
  1. lag1_logspend (~0.30–0.40) — previous month's spend is by far the most predictive single feature. Autocorrelation is the dominant pattern in monthly procurement.
  2. roll3_logspend (~0.15–0.20) — 3-month rolling average. Captures medium-term trend.
  3. aramark_boost_M (~0.10–0.15) — dollar-calibrated event signal. More precise than ordinal event_score because it's in the same unit space as the target.
  4. log_gdp / log_tourism (~0.08–0.12) — state structural features. Important for cross-state generalisation.
  5. event_score (~0.06–0.10) — ordinal event pressure index.
  6. month_sin / month_cos (~0.05) — seasonal pattern encoding.
🤔 Why Not Deep Learning?
With ~600 training observations (50 states × 12 months), deep learning (LSTM, Transformer) would overfit severely. GBM is the right tool at this data scale — it generalises well with few samples, is interpretable via feature importance, and trains in seconds. Deep learning would require years of monthly data per state to be competitive.

Q&A — Likely Questions & Strong Answers
Prepared responses for examiners and stakeholders — model decisions, data choices, limitations

Q: Why did you choose Gradient Boosting as your primary model?

  • GBM consistently achieves the lowest cross-validated RMSE on structured tabular data with mixed feature types — this is empirically established across ML competitions and academic literature (Chen & Guestrin, XGBoost 2016).
  • Our dataset has non-linear interactions: event impact × tourism revenue, lag features × event spikes. GBM handles these through tree splits without requiring explicit interaction features, unlike Ridge.
  • With ~600 training observations, GBM with max_depth=4 and subsample=0.8 generalises well — deep enough to capture patterns, shallow enough to prevent overfitting. The 5-fold CV confirmed this empirically.

Q: Why not use a neural network / LSTM for time-series forecasting?

  • LSTMs typically need 1,000+ time steps per series to learn sequential patterns. We have 12 months per state — far too few. An LSTM would memorise training data and fail to generalise.
  • For short time series with strong lag autocorrelation, classical ML outperforms deep learning (confirmed by M4/M5 forecasting competitions — GBM variants won).
  • Additionally, LSTM provides no feature importance — it's a black box. For a procurement dashboard, explainability is as important as accuracy: we need to tell the business why Indiana forecasts high in May.

Q: Your event data is manually curated — isn't that a bias risk?

  • Yes, and this is a deliberate trade-off. Automated event APIs (PredictHQ, Ticketmaster) exist but their economic impact estimates vary widely. Manual curation from verified economic studies (Indy 500 IMS report, Mardi Gras 2026 economic study, Bay Area Super Bowl Host Committee) ensures the dollar estimates are grounded in reality.
  • The risk is that we might miss events — but missing an event scores it as 0, which is conservative (predicts lower). An overestimated impact would be more dangerous. Our model's prediction corridor limits upside anyway.
  • For a production system, I would recommend integrating PredictHQ API with our manual calibration as a correction layer.

Q: How do you handle states with very little data (Vermont, North Dakota)?

  • Three mechanisms: (1) The prediction corridor clamps all forecasts within 65%–155% of historical average, preventing wild extrapolation. (2) The "organic cap" limits upside for no-event months to +15% of same month last year. (3) The aramark_boost_M for these states is near-zero (no events), so the model primarily relies on the lag features — which reflect the actual base spend accurately.
  • This is a known limitation: for states where Aramark has no or minimal contract coverage, the "historical average" itself may be unreliable as a baseline. In practice, these states represent <0.5% of total portfolio spend — their forecast uncertainty doesn't materially affect business decisions.

Q: Why predict Apr–Jun 2026 specifically? Why not the full year?

  • Forecast accuracy degrades rapidly with horizon length for monthly data — at 12 months ahead, lag features provide almost no information. Apr–Jun 2026 is 3–6 months ahead, which is within the reliable forecasting window for GBM with lag features.
  • The business reason: procurement planning for major events (FIFA WC, Indy 500, PGA Championship) requires 3–4 months lead time for supplier negotiations and inventory pre-positioning. The Apr–Jun window aligns exactly with these lead times.
  • Jan–Mar 2026 are labelled "Actual" (injected estimates from event outcomes) to provide realistic lag primes for the Apr–Jun predictions — without this, the April forecast would be based on Dec 2025 lags, which are 4 months stale.

Q: What's the difference between the "event_score" and "aramark_boost_M" features?

  • event_score (0–200): An ordinal index of general hospitality demand pressure from events. It captures all events — even those where Aramark has no venue contract — because hotel F&B, airport dining, and convention catering also benefit.
  • aramark_boost_M (dollars): A bottom-up research-backed estimate of Aramark's specific multi-channel revenue capture — venue concessions (where confirmed contracts exist) + hotel F&B + airport dining share + convention catering. This is in the same unit space as the target variable, giving the model a direct dollar-calibrated signal rather than an ordinal score.
  • Having both features allows GBM to learn that a high-scoring event without an Aramark venue contract (e.g. Coachella — no Aramark at Empire Polo Club) generates a different kind of boost than one with a confirmed contract (Indianapolis 500 — Aramark is the official concessionaire).

Q: Why log-transform the target variable?

  • Spend is log-normally distributed: California generates ~100× more spend than Vermont. Without log-transformation, the model would over-optimise for large states (minimising error on CA saves more loss than perfect performance on all small states combined).
  • Log-transformation gives each state proportionally equal training weight. The back-transformation (expm1) recovers dollar values for display.
  • Additionally, percentage-based prediction errors (which log-RMSE measures) are more meaningful for procurement planning than absolute dollar errors: "10% above forecast" is actionable regardless of state size.

Q: What are the main limitations of this model?

  • No external demand shocks: The model cannot predict recessions, pandemics, or sudden event cancellations (e.g., if FIFA WC 2026 were cancelled, all June 2026 forecasts would be dramatically wrong).
  • Spend randomisation: The "Spend Random Factor" means absolute dollar precision is impossible — we're predicting patterns in randomised values. Directional forecasts (which states grow vs shrink) are reliable; absolute magnitudes are approximations.
  • Static economic features: Population and GDP data are annual snapshots — they don't capture in-year economic changes (e.g., a Texas tech boom mid-year).
  • Segment forecast quality: For very small segments in individual states (<6 data points), the model returns "Insufficient data" — these represent long-tail contracts not captured.

Q: How would you improve this model with more time/data?

  • Multi-year historical data: 3+ years of monthly data would enable proper time series cross-validation (TimeSeriesSplit) and more robust seasonality modelling.
  • PredictHQ API integration: Automated, continuously updated event feed with verified attendance and economic impact data.
  • Property-level features: Number of rooms, property age, renovation status — these would dramatically improve Spend-per-Room prediction.
  • Hierarchical models: Instead of one model for all states, a hierarchical Bayesian model could share information across states while accounting for state-specific idiosyncrasies (e.g., Hawaii luxury resort market is fundamentally different from Iowa institutional food service).
  • Probabilistic forecast: Replace point predictions with full probability distributions using quantile regression or conformal prediction — giving stakeholders P50/P80/P90 spend scenarios for supply chain planning.

Q: Why did you build a Dash app rather than just presenting static charts?

  • Static charts require 50 state charts + 25 segment charts = 75+ slides. Interactive filtering (state dropdown, segment dropdown, state filter for segments) compresses this into 6 views that a user can navigate in 2 minutes.
  • Interactivity enables discovery: a stakeholder can ask "what happens in Hawaii in April?" and immediately see the answer without waiting for an analyst to regenerate a chart.
  • Plotly + Dash was chosen over Tableau/Power BI because: (1) it's fully code-controlled and version-manageable in git, (2) the ML integration (running Python models) is native, (3) it can be deployed as a web app without additional licensing.
